# AI Agent Workflow

이 문서는 이 저장소에서 Codex, Claude 같은 코딩 에이전트를 사용할 때의 기본 협업 규칙을 정리한다.

## 기본 원칙

- 메인 에이전트가 구현, 빌드/테스트 조율, 최종 사용자 응답을 담당한다.
- 실질적인 코드 변경이 발생하면 메인 에이전트는 필요에 따라 리뷰 전용 에이전트와 검증 전용 에이전트를 분기할 수 있다.
- 리뷰 에이전트는 읽기 전용이며, 기능적 리스크만 본다.
- 검증 에이전트는 빌드, 테스트, 스모크 체크처럼 실행 기반 검증만 담당한다.
- 최종 판단과 사용자 커뮤니케이션은 항상 메인 에이전트가 맡는다.

## 역할 분리

### 메인 에이전트

- 요구사항 해석
- 구현
- 변경 통합
- 리뷰/검증 결과 재해석
- 최종 결과 전달

### 리뷰 에이전트

- 변경된 코드와 diff 읽기
- 버그, 회귀, 데이터 손실, 호환성 문제, 의미 있는 검증 누락 탐지
- 읽기 전용 유지

리뷰 에이전트가 기본적으로 하지 않는 일:

- 스타일 취향 제안
- 과도한 리팩터링 요구
- 사소한 네이밍/문구 지적
- 코드 직접 수정

### 검증 에이전트

- 빌드 실행
- 테스트 실행
- 필요한 경우 간단한 스모크 시나리오 실행
- 실패 로그 요약 및 재현 단서 정리

검증 에이전트가 기본적으로 하지 않는 일:

- 구조 개선 제안
- 코드 리뷰
- 직접 수정

## 기본 트리거

### 리뷰 에이전트 분기

다음 조건을 모두 만족하면 기본적으로 분기한다.

1. repo-tracked 파일에 실질적인 변경이 있다.
2. 작업이 구현, 수정, 리팩터링, 버그 수정처럼 코드 변경을 포함한다.
3. 단순 설명, 문서 요약, 순수 탐색만 하는 작업이 아니다.

### 검증 에이전트 분기

다음 경우에 기본적으로 분기한다.

- 여러 파일에 걸친 변경
- 상태관리, 저장 포맷, import/export, 데이터 모델 변경
- 테스트 또는 빌드로 회귀를 빨리 잡는 편이 유리한 변경

다음 경우에는 기본적으로 생략한다.

- 단순 Q&A
- 순수 탐색
- 문서만 수정한 작업
- 사용자가 검증 실행을 원하지 않는다고 명시한 경우

## 표준 흐름

1. 메인 에이전트가 요구사항을 파악하고 구현한다.
2. 메인 에이전트가 가능한 범위에서 자체 빌드/테스트 또는 최소 점검을 수행한다.
3. 변경이 실질적이면 리뷰 에이전트를 분기한다.
4. 변경 영향이 넓거나 실행 검증이 중요하면 검증 에이전트를 추가로 분기한다.
5. 리뷰 에이전트는 기능적 리스크만 보고한다.
6. 검증 에이전트는 실행 결과와 실패 로그를 요약한다.
7. 메인 에이전트가 두 결과를 재검토하고 최종 판단을 통합해 사용자에게 전달한다.

## 결과 정리 규칙

- 리뷰 이슈가 있으면 우선순위, 영향 시나리오, 재현 조건을 포함해 짧게 정리한다.
- 리뷰 이슈가 없으면 `중요한 기능적 문제 없음`이라고 명시한다.
- 검증이 통과하면 어떤 명령 또는 시나리오가 통과했는지 짧게 적는다.
- 검증이 실패하면 실패 원인, 막힌 단계, 로그 핵심만 정리한다.
- 하위 에이전트 원문을 그대로 복붙하지 말고, 메인 에이전트가 걸러서 전달한다.

## 외부 AI 산출물 리뷰

사용자가 다른 AI가 만든 코드나 패치를 가져온 경우에도 같은 흐름을 사용한다.

1. 메인 에이전트가 변경 범위와 영향 파일을 먼저 읽는다.
2. 리뷰 에이전트를 분기해 기능적 리스크를 교차 검토한다.
3. 필요하면 검증 에이전트가 빌드/테스트를 실행한다.
4. 메인 에이전트가 최종 판단만 사용자에게 정리한다.

## 권장 포커스 순서

### 리뷰 에이전트

1. 기능 오류
2. 회귀 가능성
3. 데이터 손실/호환성 문제
4. 테스트/검증 누락

### 검증 에이전트

1. 빌드 가능 여부
2. 핵심 테스트 통과 여부
3. 변경 범위와 직접 연결된 스모크 시나리오
4. 실패 로그의 재현 단서

## 운영 메모

- 모든 작업에 무조건 리뷰/검증 분기를 거는 것은 아니다.
- 기본값은 리뷰 분기 우선, 검증 분기는 변경 영향이 넓을 때 추가하는 방식이다.
- 하위 에이전트 결과는 참고 자료이며, 최종 책임은 메인 에이전트가 진다.
- 리뷰와 검증을 한 에이전트에 합치기보다 역할을 분리하는 것을 기본값으로 한다.
